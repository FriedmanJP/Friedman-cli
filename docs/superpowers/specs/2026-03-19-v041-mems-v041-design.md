# Friedman-cli v0.4.1 — MEMs v0.4.1 Integration

## Summary

Wrap MacroEconometricModels.jl v0.4.1's constrained DSGE built-in solver fallback system. MEMs v0.4.1 removes the hard dependency on JuMP+Ipopt for constrained DSGE solving by introducing a 5-tier solver hierarchy (NonlinearSolve → Projected Newton → Optim.jl → NLopt.jl → JuMP+Ipopt/PATH). This integration adds a `--constraint-solver` CLI option to 4 DSGE commands, supports a new `NonlinearConstraint` type via TOML config, and bumps the MEMs compat.

## Upstream Changes (MEMs v0.4.1)

### 5-Tier Solver Hierarchy

| Tier | Backend | Handles |
|------|---------|---------|
| 1 | NonlinearSolve.jl | Unconstrained, non-binding box |
| 2 | Projected Newton | Box-constrained perfect foresight |
| 3 | Optim.jl `Fminbox(LBFGS())` | Box-constrained steady state |
| 4 | NLopt.jl `LD_SLSQP` | Nonlinear inequality constraints |
| 5 | JuMP+Ipopt / JuMP+PATH | Explicit override (`solver=:ipopt`/`:path`) |

### Changed Function Signatures

- `perfect_foresight(spec; ..., solver=nothing)` — new `solver` keyword
- `compute_steady_state(spec; method=:default)` — solver auto-selected internally
- `solve(spec; ..., solver=nothing)` — new `solver` keyword when constraints present

### New Types & Constructors

- `NonlinearConstraint{T}` — general inequality constraint type
- `nonlinear_constraint(fn; label="...")` — constructor
- `variable_bound(:var, lower=0.0, upper=1.0)` — now supports both lower and upper

### New Dependencies (transitive through MEMs)

- NLopt.jl (LGPL) — no new direct deps needed in Friedman-cli
- ForwardDiff.jl (MIT) — no new direct deps needed in Friedman-cli

## Design

### 1. `--constraint-solver` Option on 4 DSGE Commands

**Commands:** `dsge solve`, `dsge steady-state`, `dsge perfect-foresight`, `dsge bayes estimate`

```
Option("constraint-solver", ' ', String, "")  # nonlinearsolve|optim|nlopt|ipopt|path
```

Named `--constraint-solver` (not `--solver`) to avoid collision with the existing `--solver` option on `dsge bayes estimate` which controls the DSGE solution method (gensys/klein/perturbation). The hyphen is converted to underscore by the parser, so handlers receive `constraint_solver` kwarg.

- When empty (default): MEMs auto-selects per 5-tier hierarchy
- When set: passed as `solver=Symbol(constraint_solver)` kwarg to the upstream function
- **Validation:** reject values not in `["nonlinearsolve", "optim", "nlopt", "ipopt", "path"]` with a clear error message
- **Not added to:** OccBin commands (`occbin_solve`/`occbin_irf`) — these use piecewise-linear regime switching on a separate code path

**Handler changes:**

In `_dsge_solve()`, `_dsge_steady_state()`, `_dsge_perfect_foresight()`, `_dsge_bayes_estimate()`:
```julia
solver_kw = isempty(constraint_solver) ? (;) : (; solver=Symbol(constraint_solver))
sol = solve(spec; method=Symbol(method), constraints=constraints, solver_kw...)
```

**`_solve_dsge()` helper** in shared.jl must also be updated to accept and forward the `constraint_solver` kwarg, since `dsge solve` delegates to this helper.

### 2. NonlinearConstraint via TOML Config

**New TOML section** alongside existing `[[constraints.bounds]]`:

```toml
# Existing box constraints (route to OccBin — unchanged)
[[constraints.bounds]]
variable = "i"
lower = 0.0

# New nonlinear constraints (route to new solver hierarchy)
[[constraints.nonlinear]]
expr = "K[t] + C[t] <= Y[t]"
label = "resource constraint"
```

**Constraint routing:**

- `[[constraints.bounds]]` without `--constraint-solver` → routes to OccBin (preserves current behavior exactly)
- `[[constraints.bounds]]` with `--constraint-solver` set → routes to the new solver hierarchy (box constraints via Optim.jl/NLopt.jl/Ipopt)
- `[[constraints.nonlinear]]` → always routes to the new solver hierarchy
- Mixed `bounds` + `nonlinear` is allowed — all constraints passed to the new solver hierarchy together

**Parsing:** Delegate to MEMs' existing `parse_constraint(expr, spec)` function, which already handles expression parsing. No custom parser needed in Friedman-cli.

**Implementation in `_load_dsge_constraints()`** (shared.jl):

The function signature becomes `_load_dsge_constraints(path; spec=nothing)`. When `spec` is `nothing` and nonlinear constraints are found, error with a clear message.

```julia
function _load_dsge_constraints(path; spec=nothing)
    config = load_config(path)
    cons_cfg = get_dsge_constraints(config)

    has_bounds = haskey(cons_cfg, "bounds") && !isempty(cons_cfg["bounds"])
    has_nonlinear = haskey(cons_cfg, "nonlinear") && !isempty(cons_cfg["nonlinear"])

    if has_nonlinear && spec === nothing
        error("nonlinear constraints require a loaded DSGE spec")
    end

    constraints = Any[]
    if has_bounds
        append!(constraints, _parse_occbin_constraints(cons_cfg["bounds"]))
    end
    if has_nonlinear
        append!(constraints, _parse_nonlinear_constraints(cons_cfg["nonlinear"], spec))
    end
    return constraints
end
```

**Return type:** `Vector{Any}` containing `OccBinConstraint` and/or `NonlinearConstraint` elements. Downstream handler routing is determined by `--constraint-solver` flag, not by constraint type.

**Config parsing in `get_dsge_constraints()`** (config.jl):

Currently returns the raw TOML dict for `bounds`. Extend to also return `nonlinear`:

```julia
function get_dsge_constraints(config)
    result = Dict{String, Any}()
    if haskey(config, "constraints")
        if haskey(config["constraints"], "bounds")
            result["bounds"] = config["constraints"]["bounds"]
        end
        if haskey(config["constraints"], "nonlinear")
            result["nonlinear"] = config["constraints"]["nonlinear"]
        end
    end
    return result
end
```

### 3. Handler Routing Logic

In `dsge.jl` handlers that accept `--constraints`, the routing changes:

```julia
cons = _load_dsge_constraints(constraints; spec=spec)
if !isempty(cons) && isempty(constraint_solver)
    # No --constraint-solver set → OccBin path (backward-compatible default)
    occbin_cons = filter(c -> c isa OccBinConstraint, cons)
    ob_sol = occbin_solve(spec, occbin_cons; ...)
else
    # --constraint-solver set → new solver hierarchy path
    solver_kw = isempty(constraint_solver) ? (;) : (; solver=Symbol(constraint_solver))
    sol = solve(spec; constraints=cons, solver_kw...)
end
```

This keeps the OccBin path as the default when no `--constraint-solver` is specified, preserving full backward compatibility. When `--constraint-solver` is set, all constraints (bounds + nonlinear) route through the new solver hierarchy.
